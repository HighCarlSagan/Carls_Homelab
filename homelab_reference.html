2
Clone repo and run restore.sh
git clone git@github.com:HighCarlSagan/Carls_Homelab.git && bash Carls_Homelab/restore.sh
3

users.yml, Cloudflare token, tunnel credentials JSON, MQTT password, Pi-hole password.

5

⚠ Estimated recovery time: 30–45 minutes. Keep secrets backed up in Bitwarden as secure notes. The daily backup archive at files.highcarlsagan.dev/files/backups/ contains config snapshots — download before the Pi dies if possible.

05 // CRITICAL NOTES

Gotchas & Deviations from Blueprint

⛔
Caddy — DO NOT reinstall from apt. Custom xcaddy build with patched caddy-dns/cloudflare module. Upstream regex {35,50} rejects Cloudflare's newer cfut_ tokens (52 chars). Patched to {35,100}. Source: ~/caddy-cloudflare-patched/. Rebuild: cd ~/caddy-cloudflare-patched && ~/go/bin/xcaddy build --with github.com/caddy-dns/cloudflare=.
⛔
Never use nano on Pi. TERM=xterm-kitty causes "Error opening terminal". Always use sed for config edits. For TUI-based tools, SSH from xterm or alacritty with TERM=xterm-256color.
⚠
Authelia apt repo is dead. apt.authelia.com returned 404 — removed from sources.list.d. Authelia installed from GitHub binary. Do not attempt apt install authelia.
⚠
Pi-hole — always use Docker. Native installer has hardcoded ncurses dialogs incompatible with Kitty terminal. Pi-hole v6 renamed commands: -w → pihole allow, WEB_PORT → FTLCONF_webserver_port, DNSMASQ_LISTENING → FTLCONF_dns_listeningMode.
⚠
Mosquitto passwd ownership. After creating/recreating passwd file with mosquitto_passwd, always run: sudo chown mosquitto:mosquitto /etc/mosquitto/passwd. File created as root — Mosquitto runs as its own user and can't read it.
⚠
Home Assistant reverse proxy config. Must be at root level of configuration.yaml, NOT nested under homeassistant:. Correct: http: / use_x_forwarded_for: true / trusted_proxies: [127.0.0.1, ::1]
ℹ
Subnet is 192.168.0.x not 192.168.1.x as in blueprint. Hostname is 4pi.local not passive.local. Substitute everywhere when following blueprint instructions.
ℹ
Cloudflare tunnel routing. Originally routed directly to backend ports — fixed by routing all ingress through Caddy HTTPS:443 with originServerName + noTLSVerify: true per subdomain in /etc/cloudflared/config.yml.
ℹ
Uptime Kuma monitors. Use http://127.0.0.1:PORT not https://localhost:PORT — services run plain HTTP internally and localhost resolves to IPv6 (::1) which causes ECONNREFUSED.

10 // EXTENDING

Adding a New Service / Page

Every new service follows the same 5-step pattern: install → pick a port → add to Caddyfile → add Cloudflare CNAME → add cloudflared ingress rule.

1
Install the service and pick a free port
Check the port map above to avoid conflicts. Recommended free ports: 8082, 8083, 8090, 3002, 4000+. Run the service bound to 0.0.0.0 or 127.0.0.1 on your chosen port.
2
Add entry to Caddyfile
Edit /etc/caddy/Caddyfile using sed. Add a new block — include import authelia to protect it, or omit for a public page.
# Pattern for a PROTECTED service (requires Authelia login) myapp.highcarlsagan.dev { import tls_cloudflare import authelia reverse_proxy http://127.0.0.1:PORT } # Pattern for a PUBLIC page (no auth — use carefully) myapp.highcarlsagan.dev { import tls_cloudflare reverse_proxy http://127.0.0.1:PORT } # Reload Caddy after editing sudo systemctl reload caddy
3
Add Cloudflare DNS CNAME record
Cloudflare dashboard → highcarlsagan.dev → DNS → Add record.
Type: CNAME · Name: myapp · Target: 344e646f-5867-47ad-88dd-1ee169051622.cfargotunnel.com · Proxy: ON
4
Add ingress rule to cloudflared config
Edit /etc/cloudflared/config.yml — add before the catch-all http_status:404 line.
# Add this block to /etc/cloudflared/config.yml ingress section - hostname: myapp.highcarlsagan.dev service: https://localhost:443 originRequest: noTLSVerify: true originServerName: myapp.highcarlsagan.dev # Restart cloudflared after editing sudo systemctl restart cloudflared
5
Add monitor to Uptime Kuma + commit to git
status.highcarlsagan.dev → Add Monitor → HTTP → http://127.0.0.1:PORT. Then update the git repo.
# Commit updated configs to git cd ~/homelab-configs sudo cp /etc/caddy/Caddyfile caddy/ sudo cp /etc/cloudflared/config.yml cloudflared/ git add -A git commit -m "add: myapp service on port PORT" git push
ℹ For a static HTML page (like this reference doc): drop the file in /srv/filebrowser/ and it's immediately accessible via Filebrowser. To give it a clean subdomain, point Caddy to file_server { root /srv/filebrowser } or serve it from a simple Python server on a free port.

11 // IOT

Adding ESP32 Devices

ESP32 devices communicate with Home Assistant via Mosquitto MQTT using HA's auto-discovery protocol. When configured correctly, HA automatically creates the device and all its entities — no YAML editing needed.

MQTT Broker
192.168.0.100
Port: 1883 · User: mqttuser
Discovery Prefix
homeassistant/
HA listens on this prefix for auto-discovery
1
Flash ESP32 with firmware
Use PlatformIO or Arduino IDE. Add library: knolleary/PubSubClient. Set WiFi credentials, MQTT broker IP, device ID (unique per device), and MQTT password.
2
Publish MQTT discovery payload on boot
On connect, publish a JSON config message to homeassistant/sensor/DEVICE_ID_sensorname/config with retain: true. This tells HA what the device is and what topics to subscribe to.
// Minimal discovery payload (publish once on connect, retained) // Topic: homeassistant/sensor/esp32_node01_temp/config { "name": "Temperature", "uniq_id": "esp32_node01_temp", "stat_t": "homelab/esp32_node01/temperature", // where readings are published "unit_of_meas": "°C", "dev_cla": "temperature", "dev": { "ids": ["esp32_node01"], "name": "Workshop Node 01", "mdl": "ESP32", "mf": "Espressif" } }
3
HA auto-discovers the device
After the ESP32 boots and publishes discovery, go to HA → Settings → Devices & Services → Devices. The device appears automatically with all entities. No YAML editing needed.
4
Publish sensor readings periodically
Publish plain values (e.g. "23.4") to the state topic defined in discovery. HA reads and stores them automatically.
// Publish a sensor reading every 30 seconds // Topic: homelab/esp32_node01/temperature // Payload: "23.4" (plain string, not JSON) // For a switch/relay, add a command topic to discovery: "cmd_t": "homelab/esp32_node01/relay/set" // HA sends "ON" or "OFF" here // Subscribe to cmd_t in firmware and toggle GPIO accordingly
5
Add device to Uptime Kuma (optional)
Add a TCP monitor to DEVICE_LAN_IP:80 if the ESP32 runs a web server, or monitor via MQTT topic freshness using a HA automation that triggers an alert if the device goes silent.
⚠ Device ID must be unique per ESP32. Use a naming convention like esp32_location_number (e.g. esp32_workshop_01, esp32_desk_01). Duplicate IDs cause HA to merge two devices into one and break entity tracking.
ℹ Full firmware template is in the git repo at Carls_Homelab/ (to be added). It includes WiFi reconnect logic, MQTT reconnect with backoff, discovery for sensor + relay, and OTA update support.
