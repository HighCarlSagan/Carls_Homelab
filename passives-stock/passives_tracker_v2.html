STORAGE GUIDE
SYSTEM DESIGN

📒 SMD COMPONENT BOOK PRIMARY

Tape-strip books (100–200 pockets) for 0603/0805 resistors and MLCCs. Each pocket holds one value on its original strip.

Best for passives you grab constantly. Visual, compact, portable to bench.

₹300–600 / book~120 values

🗄️ MODULAR DRAWER UNIT PRIMARY

64–80 drawer unit (Duratool or local equivalent). Label each drawer. Use for electrolytics, 1210+ passives, and anything too large for a book.

₹1500–350064–80 compartments

📦 ESD GRID BOX

For active reels and opened cut-tape on current projects. One box per active board keeps you from mixing stock.

₹200–500 / box~20 part types

🤐 LABELLED ZIP BAGS

A4 binder with zip pouches for DigiKey bulk overflow. Write value + PN on bag immediately on receipt — before the reel label disappears.

₹5–10 / bagUnlimited
SORT ORDER
  • Resistors 0603: E24 ascending (10Ω → 1MΩ) — Book 1
  • Resistors 0805: Power → value ascending — Book 2 or Drawer Row A
  • MLCC C0G: pF ascending — Book 3 left half
  • MLCC X7R: nF/µF ascending — Book 3 right half
  • MLCC X5R: µF ascending — Book 4 or 0805 Drawer
  • Inductors: Sort by type first (Power → CMC → Ferrite), then value
  • Crystals: Frequency ascending, SMD before TH — small drawer or anti-static tube
  • Electrolytics TH: Voltage first (35V → 63V → 100V) then value — Drawer unit bottom
TIPS FOR BANGALORE
  • 🌧️ Humidity: Silica gel in every drawer unit. Swap every 6 months. Electrolytic pads corrode faster than you'd expect.
  • ⚡ Reform old caps: Electrolytics sitting unused 2+ years — apply rated voltage through 100Ω for 30 min before circuit use.
  • 🏷️ Label immediately: Write DigiKey PN on every bag on receipt. Once the reel label is off, the value is lost forever.
  • 🛒 DigiKey bulk strategy: Buy 1000-pcs reels for 100nF, 10nF, 1µF X7R and 1kΩ, 10kΩ resistors. Cut tape everything else until you know burn rate.
  • 🧲 Inductors: Power inductors are DCR-sensitive — verify DCR against current limit before stocking bulk. Coilcraft SER / Bourns SRR series are reliable.

STEP 1 FILE SETUP

Create the working directory and place both files in it. Keep them together — the server looks for the HTML in the same folder as itself.

mkdir -p ~/passives-stock
cd ~/passives-stock
# copy server.py and passives_tracker_v2.html here
ls -lh
# should show: server.py  passives_tracker_v2.html
state.json and the backups/ folder are created automatically on first save. You don't need to create them.

STEP 2 FIRST RUN — MANUAL TEST

Before setting up auto-start, verify everything works manually.

cd ~/passives-stock
python3 server.py

The terminal will print your LAN IP:

  Local:    http://localhost:8080
  Network:  http://192.168.1.42:8080   ← your actual IP
  State:    ~/passives-stock/state.json

Open http://localhost:8080 in your browser. The ● SAVED indicator top-right confirms the server is writing state. Make a change, stop the server with Ctrl+C, restart it, and verify the change persisted.

If port 8080 is already in use: run python3 server.py --port 9090 and update the systemd service file accordingly.

STEP 3 STATIC IP — TWO APPROACHES

Option A — DHCP reservation on your router (preferred if accessible)

Log into your router admin page (usually 192.168.1.1 or 192.168.0.1). Look for DHCP / LAN / Connected Devices. Find your PC's MAC address, and bind it to a fixed IP like 192.168.1.100. ISP routers in India vary — if yours doesn't expose this, use Option B.

# Find your MAC address and current IP
ip link show          # look for your ethernet/wifi interface
ip addr show          # shows current IP next to inet

Option B — Static IP on the machine itself via NetworkManager (works regardless of router)

This sets the IP directly on your Arch machine. Replace enp3s0 with your actual interface name from ip link show, and choose an IP outside your router's DHCP range (e.g. 192.168.1.200).

# Check your interface name first
ip link show

# Set a static IP (adjust interface, IP, and gateway to match your network)
nmcli con mod "$(nmcli -g name con show --active)" \
  ipv4.addresses "192.168.1.200/24" \
  ipv4.gateway "192.168.1.1" \
  ipv4.dns "192.168.1.1,8.8.8.8" \
  ipv4.method manual

# Restart the connection to apply
nmcli con down "$(nmcli -g name con show --active)" && \
nmcli con up   "$(nmcli -g name con show --active)"

# Verify
ip addr show
If you use systemd-networkd instead of NetworkManager: check with systemctl status NetworkManager systemd-networkd — whichever is active is your network manager. On a standard Arch + Hyprland setup it's usually NetworkManager.
Once set: your server will always be reachable at that IP, even after reboots. Write it down — you'll need it for the /etc/hosts step.

STEP 4 passive.local — mDNS VIA AVAHI

Avahi broadcasts your machine's hostname over mDNS so every device on the LAN can reach it by name with zero client configuration. iOS, Android, macOS, and Linux all support mDNS natively.

Install and enable

sudo pacman -S avahi nss-mdns

# Set your machine hostname to 'passive'
sudo hostnamectl set-hostname passive

# Enable mDNS resolution in nsswitch.conf
# Open the file and find the 'hosts:' line:
sudo nano /etc/nsswitch.conf
# Change:  hosts: mymachines resolve [!UNAVAIL=return] files myhostname dns
# To:      hosts: mymachines mdns_minimal [NOTFOUND=return] resolve [!UNAVAIL=return] files myhostname dns
# (add 'mdns_minimal [NOTFOUND=return]' right after 'mymachines')

# Enable and start avahi
sudo systemctl enable --now avahi-daemon

Verify it's working

# From your own machine
ping passive.local

# From another machine on the LAN
ping passive.local
avahi-resolve -n passive.local   # if avahi-utils installed
Result: any browser on your LAN can now open http://passive.local:8080 — no config on the client device needed.
Hostname change note: changing the hostname takes effect on next login/reboot for the shell prompt. Avahi picks it up immediately. If you were using your old hostname for SSH or anything else, update those references.

STEP 5 /etc/hosts — CUSTOM DOMAIN FALLBACK

If you want passive.stock specifically (instead of passive.local), or want a fallback that works without Avahi, add entries to /etc/hosts on each device. Replace 192.168.1.200 with your actual static IP.

Linux / Arch (your PC)

sudo nano /etc/hosts
# Add this line:
192.168.1.200   passive.stock passive.local

Android (phone)

# Android does not expose /etc/hosts without root.
# Use the IP directly: http://192.168.1.200:8080
# Or use passive.local via mDNS (Step 4) — works on Android 12+
# in Chrome and most browsers without any config.

iOS / iPadOS

# iOS does not allow /etc/hosts edits without a DNS config profile.
# Use passive.local (mDNS, Step 4) — Safari supports it natively.
# Or bookmark the IP directly: http://192.168.1.200:8080
Bottom line: passive.local (Avahi/mDNS) is the zero-hassle option for phone and iPad. The /etc/hosts approach is most useful on Linux machines where you want the custom .stock domain.

STEP 6 AUTO-START — SYSTEMD USER SERVICE

This starts the server when you log in, restarts it if it crashes, and logs to journald.

Install the service

# Edit the service file first — update the two paths to match yours
nano ~/passives-stock/passives-stock.service
# Change /home/mak → your actual home directory

# Install and enable
mkdir -p ~/.config/systemd/user/
cp ~/passives-stock/passives-stock.service ~/.config/systemd/user/
systemctl --user daemon-reload
systemctl --user enable --now passives-stock.service

Verify it's running

systemctl --user status passives-stock.service
# Should show: Active: active (running)

# Live logs
journalctl --user -u passives-stock.service -f

# Test that it survives a reboot
sudo reboot
# After login: systemctl --user status passives-stock.service

Manage the service

systemctl --user stop    passives-stock.service   # stop now
systemctl --user start   passives-stock.service   # start now
systemctl --user restart passives-stock.service   # restart (after editing HTML)
systemctl --user disable passives-stock.service   # stop auto-starting
After editing the HTML you do not need to restart the server — the server reads the file fresh on each request. Just reload the browser tab.

STEP 7 ACCESSING FROM ALL DEVICES

Once the server is running and Avahi is enabled, all four devices work the same way:

ARCH PC
http://passive.local:8080
or http://localhost:8080
RASPBERRY PI (future)
http://passive.local:8080
same URL once migrated
iPHONE / iPAD
http://passive.local:8080
Safari — mDNS works natively
ANDROID PHONE
http://passive.local:8080
Chrome — mDNS on Android 12+
All devices must be on the same WiFi/LAN. This does not work over mobile data or from outside your home network — it's LAN-only by design.
Multi-device editing: changes from any device save to the same state.json. Last write wins. Since you're the only user this is fine — avoid editing from two devices at exactly the same moment.

STEP 8 FIREWALL NOTES

You currently have no active firewall, so port 8080 is open by default. This section is here for when you add one later.

Check if a firewall is active

systemctl status nftables ufw firewalld 2>/dev/null | grep -E "Active|Loaded"
# If all show 'inactive' or 'not-found' — no firewall running, nothing to do.

If you add nftables later

# Allow port 8080 from LAN (adjust interface — usually enp3s0 or wlan0)
sudo nft add rule inet filter input \
  iifname "enp3s0" tcp dport 8080 accept

# Also allow mDNS (UDP 5353) for Avahi
sudo nft add rule inet filter input udp dport 5353 accept

# Make rules persistent (save to your nftables config)
sudo nft list ruleset | sudo tee /etc/nftables.conf

If you add ufw later

sudo ufw allow 8080/tcp
sudo ufw allow 5353/udp   # mDNS for Avahi
Avahi and mDNS use UDP port 5353 and multicast address 224.0.0.251. If passive.local resolves on your machine but not from other devices, a firewall blocking 5353 is the most likely cause.

STEP 9 TROUBLESHOOTING

● OFFLINE shown in header — server not reached

# Check if server is running
systemctl --user status passives-stock.service
curl http://localhost:8080/health

# Check what's on port 8080
ss -tlnp | grep 8080

passive.local not resolving from this machine

# Verify avahi is running
systemctl status avahi-daemon

# Verify nsswitch.conf has mdns_minimal
grep ^hosts /etc/nsswitch.conf

# Test resolution directly
avahi-resolve -n passive.local
getent hosts passive.local

passive.local not resolving from phone/iPad

# Verify both devices are on the same WiFi network (not guest VLAN)
# Some routers isolate devices on guest networks — mDNS won't cross VLANs

# Try the IP directly to rule out DNS as the issue
# http://192.168.1.200:8080 (your static IP)

# Check Avahi is advertising on the right interface
avahi-browse -a -t | head -20

state.json not updating / saves not persisting

# Check server logs for write errors
journalctl --user -u passives-stock.service -n 50

# Check file permissions
ls -lh ~/passives-stock/state.json

# Check disk space
df -h ~/

Port 8080 conflict

# Find what's using 8080
ss -tlnp | grep 8080
fuser 8080/tcp

# Switch to a different port — edit ExecStart in the service file
nano ~/.config/systemd/user/passives-stock.service
# Change: --port 8080  →  --port 9090
systemctl --user daemon-reload
systemctl --user restart passives-stock.service

Service won't start after reboot

# Check if lingering is enabled (needed for headless / pre-login start)
loginctl show-user $USER | grep Linger
# If Linger=no and you want it to start before login:
sudo loginctl enable-linger $USER

# View full service failure log
journalctl --user -u passives-stock.service --since "1 hour ago"

STEP 10 MIGRATING TO RASPBERRY PI

When your Pi network host is ready, migration is a single rsync command. Your entire state — all stock counts, custom rows, controllers — moves with the files.

1. Copy everything to the Pi

# From your Arch machine (replace pi-ip with Pi's IP, pi with Pi username)
rsync -av ~/passives-stock/ pi@<pi-ip>:~/passives-stock/
# This copies server.py, HTML, state.json, and backups/ — everything.

2. Set up Python on the Pi (if needed)

# Python 3 is pre-installed on Raspberry Pi OS
python3 --version
# No pip installs needed — server uses stdlib only

3. Set up Avahi on the Pi

# Raspberry Pi OS (Bookworm/Bullseye)
sudo apt install -y avahi-daemon libnss-mdns

# Set the hostname to 'passive' (same as your Arch machine)
sudo hostnamectl set-hostname passive
sudo nano /etc/hosts
# Update the line that says: 127.0.1.1  raspberrypi
# Change it to:            127.0.1.1  passive

sudo systemctl enable --now avahi-daemon

4. Set up systemd user service on the Pi

# Edit service file — update /home/mak → /home/pi
nano ~/passives-stock/passives-stock.service

mkdir -p ~/.config/systemd/user/
cp ~/passives-stock/passives-stock.service ~/.config/systemd/user/
systemctl --user daemon-reload
systemctl --user enable --now passives-stock.service

# Enable lingering so it starts without login (headless Pi)
sudo loginctl enable-linger pi

5. Stop the service on your Arch machine

# On your Arch PC — disable so both aren't running at once
systemctl --user disable --now passives-stock.service
# The URL stays the same: http://passive.local:8080
# It now resolves to the Pi instead.
Both machines can't use the same hostname simultaneously. Once the Pi takes over as passive, disable Avahi on your Arch machine or rename it: sudo hostnamectl set-hostname mak-arch (or whatever you prefer).
Sync state back from Pi to Arch if you ever want a local backup: rsync -av pi@<pi-ip>:~/passives-stock/state.json ~/passives-stock/
